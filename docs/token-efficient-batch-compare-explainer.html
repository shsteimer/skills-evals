Turn the pipeline mechanical first, analytical last.

This branch keeps Sean’s comparison-tooling direction intact, then narrows the expensive LLM work to a single focused batch-compare stage. Run-level and batch-level analysis become scripted artifacts with explicit focus handoff.

Phase 1: `run-report.json` Phase 2: `batch-focus.json` Phase 3: `comparison-focus.json` LLM reserved for compare
What changed on this branch

Checks now resolve from the result-folder criteria snapshot. Batch summarization is scripted. Comparison artifacts now emit deterministic focus data, and the assembly step preserves that focus after the final recommendation pass.

What the benchmark proves

The scripted path runs quickly and reliably on replayed historical batches. The February data is sparse, so it validates pipeline behavior and runtime more than it validates focus reduction quality.

Pipeline Map

Three phases, one narrower LLM boundary

Sean’s shared platform + this branch’s gating/focus layer
Phase 1

Assemble run reports

  • `assemble-eval.js` now emits `run-report.json` beside judged eval output.
  • `resolve-checks.js` reads `criteria.txt` from the result folder, not the live task.
  • Checks, warnings, token usage, diff shape, and activity summaries become deterministic data.
Phase 2

Script the batch summary

  • `summarize-batch.js` prefers `run-report.json` and falls back to `eval-result.json` for older batches.
  • It emits `batch-summary.json`, `batch-summary-data.js`, and `batch-focus.json`.
  • Common failures, warnings, unresolved criteria, timeout-heavy groups, and high variance become focus signals.
Phase 3

Focus the compare stage

  • `compare-batches.js` merges batch focus inheritance with delta-based reasons.
  • `comparison-focus.json` tells the analytical pass exactly which groups and runs deserve inspection.
  • `assemble-comparison.js` preserves scripted focus while merging recommendation output.
End State

One required LLM stage

  • Run-level analysis is mechanical.
  • Batch-level analysis is mechanical.
  • The LLM is saved for thoughtful batch comparison, not repetitive bookkeeping.
Implementation Lens

What this branch adds on top of the parent comparison work

Phase 1 → Phase 3 stitched into operator flow

Deterministic run contract

  • New `scripts/assemble-run-report.js` standardizes run-level evidence.
  • Warnings such as `timed-out`, `no-commits`, and `no-diff` are explicit instead of inferred later.
  • Run artifacts can feed future scripted viewers without re-parsing raw logs.

Backward-compatible summary path

  • Older batches still summarize through `eval-result.json` fallback.
  • Newer batches automatically expose `batch-focus.json` for the compare stage.
  • The branch hardens the pipeline without forcing immediate migration of old data.

Focused comparison handoff

  • Per-group deltas can trigger `score-*`, `success-*`, `token-*`, `duration-*`, and `stability-regression` reasons.
  • Inherited focus reasons are labeled by source: `baseline:*` and `candidate:*`.
  • The compare skill now tells the model to start from focused runs, not from the entire batch.
Important fix folded into Phase 3

The original comparison assembly step overwrote `comparison.analysis`, which would have erased `comparison-focus.json` once the subagent recommendation was merged. This branch preserves the scripted focus and appends the recommendation fields on top.

How to interpret the benchmark honestly

Good mechanics, incomplete dataset

Confirmed

  • The new branch can replay older batches without requiring fresh agent runs.
  • The scripted artifacts write cleanly and fast.
  • The compare stage now has a deterministic focus contract to hand to the final LLM pass.
  • The branch remains fully green under test and lint.

Still worth testing next

  • Run a fresh modern batch with checks, commits, and richer run metadata.
  • Measure focus reduction on a dataset where not every run is mechanically suspicious.
  • Compare “all runs inspected” vs “focus-only inspected” on token spend once the final compare agent is exercised.
Recommended next benchmark

Generate two fresh batches from the current pipeline, summarize both with `run-report.json` present, then run `compare-batches` and count `focusRuns / total runs`. That will give a real token-efficiency read rather than a compatibility replay.
